<task type="auto">
  <name>Task 1: Validate parallel execution with pytest-xdist</name>
  <files></files>
  <action>
Run the full test suite with parallel execution enabled:

1. Run `pytest tests/ -n auto -v --timeout=30` to distribute tests across CPU cores.
2. Run the same command a second time to verify deterministic results.
3. If any tests fail under parallel execution but pass sequentially (`pytest tests/ -v`):
   - The failure is a shared-state issue. Diagnose which state is leaking.
   - Common culprits: hardcoded file paths (check that tests use `tmp_path`), module-level mutable state beyond the three known singletons, or test ordering dependencies.
   - Fix by adding to the autouse `_reset_singletons` fixture in conftest.py or by making the specific test use `tmp_path`.
   - Do NOT disable xdist to fix -- find and fix the root cause.

4. Verify xdist worker count: output should show `[gw0]`, `[gw1]`, etc. indicating multiple workers.

Key details:
- pytest-xdist gives each worker its own subprocess with its own Python process (and thus its own module-level state). The autouse singleton reset from Plan 01 ensures clean state within each worker's sequential test execution.
- Tests using `tmp_path` already get worker-isolated temporary directories.
- test_tui.py tests may need special attention -- Textual headless mode should work in xdist workers since each gets its own event loop, but if failures occur, consider marking them with `@pytest.mark.xdist_group("tui")` to keep them on the same worker.
  </action>
  <verify>
`pytest tests/ -n auto -v --timeout=30` exits 0. Running it twice produces the same pass count. Output shows multiple workers (gw0, gw1, etc.).
  </verify>
  <done>Full test suite passes under `pytest -n auto` with tests distributed across CPU cores. No shared-state failures. Deterministic results on repeated runs.</done>

<task type="auto">
  <name>Task 2: Set up VCR.py cassette infrastructure and validate recording</name>
  <files>tests/cassettes/.gitkeep, tests/conftest.py</files>
  <action>
1. Create the `tests/cassettes/` directory with a `.gitkeep` file so the directory is tracked in git.

2. The `vcr_config` fixture was already added in Plan 01's conftest.py. Verify it exists and has:
   - `filter_headers`: ["authorization", "x-api-key", "x-goog-api-key", "anthropic-api-key"]
   - `record_mode`: "none" (CI replay mode; use `--record-mode=once` locally to create cassettes)
   - `cassette_library_dir`: "tests/cassettes"
   - `decode_compressed_response`: True

3. If `vcr_config` fixture is NOT already in conftest.py (Plan 01 may have deferred it), add it now as a module-scoped fixture.

4. Validate VCR.py + pytest-recording integration by running:
   ```bash
   pytest --co -q -k "not test_" --fixtures 2>&1 | grep vcr
   ```
   This should show the `vcr_config` and `vcr` fixtures are registered.

5. Create a minimal smoke test to validate the VCR infrastructure works. Add a test function to a new or existing test file (or run inline) that uses `@pytest.mark.vcr` decorator. Since there's no cassette recorded yet, the test should demonstrate that VCR.py properly intercepts and that `--record-mode` flag is recognized:
   ```bash
   pytest --record-mode=none --co -q 2>&1 | head -5
   ```
   This verifies pytest-recording CLI integration works.

Key details:
- VCR.py 8.1.1 patches httpcore (not httpx directly). google-genai uses httpx which delegates to httpcore, so cassette recording/replay works for REST API calls (generate_content, models.list, etc.).
- WebSocket-based Live API calls (GeminiSession) CANNOT be recorded by VCR.py. Those need mock patterns (Phase 8 scope).
- Actual cassette recording happens in Phase 8 when integration tests are written. This plan only establishes the infrastructure.
- `record_mode: "none"` means tests that use `@pytest.mark.vcr` will fail if no cassette exists. This is intentional for CI -- cassettes must be committed to git.
  </action>
  <verify>
`tests/cassettes/.gitkeep` exists. `pytest --fixtures -q 2>&1 | grep -E "vcr_config|vcr"` shows VCR fixtures registered. `pytest --record-mode=none --co -q` runs without errors (flag is recognized).
  </verify>
